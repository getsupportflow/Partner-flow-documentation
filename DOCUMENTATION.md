# PartnerFlow Documentation Guide

This file is the authoritative reference for how this documentation site is structured and how to maintain or extend it. Follow these conventions when adding, editing, or generating documentation — whether you are a human author or an AI agent.

---

## Repository Layout

```
Partner-flow-documentation/    ← Docusaurus site root AND repo root
├── openapi.yaml               ← Auto-updated by CI from the main app repo; do not edit manually
├── DOCUMENTATION.md           ← This file
├── docusaurus.config.ts       ← Site configuration, plugin registration
├── sidebars.ts                ← Autogenerated sidebar (do not manually add doc IDs here)
├── package.json
├── src/                       ← Custom React components and pages
├── static/                    ← Static assets (images, favicon)
└── docs/                      ← All user-facing content lives here
    ├── index.md               ← Landing page
    ├── getting-started/       ← Onboarding guides
    ├── create-service/        ← Service flow builder guides
    ├── sharing-service/       ← Partner sharing guides
    ├── user-guides/
    │   ├── admin/             ← Admin and manufacturer guides
    │   │   └── integrations/
    │   └── non-admin/         ← Service provider guides
    ├── api/                   ← AUTO-GENERATED at deploy time — gitignored, do not commit or edit manually
    ├── faq.md
    ├── release-notes.md
    └── support.md
```

## File and Folder Naming Conventions

- Use **kebab-case** for all file and folder names: `service-options.md`, `gmail-integration.md`
- Use `.md` for standard content, `.mdx` only when the page uses React components or JSX
- Never use spaces in file or folder names
- Folder names should be short and descriptive: `integrations/`, not `integration-guides/`

## Required Frontmatter

Every content file must include this frontmatter block at the top:

```yaml
---
title: Human-Readable Page Title
description: One sentence describing what this page covers (used for SEO meta and sidebar).
sidebar_position: N
---
```

- `title` — shown in the browser tab and as the page heading in search results
- `description` — keep it to one clear sentence; do not repeat the title
- `sidebar_position` — integer controlling order within the folder; `1` is first

Optional frontmatter:

```yaml
---
slug: /custom-url-path        # Override the auto-generated URL
keywords: [keyword1, keyword2] # Improve search discoverability
---
```

## Adding a New Page

1. Create a `.md` file in the appropriate folder
2. Add the required frontmatter (title, description, sidebar_position)
3. Write the content following the style guide below
4. The sidebar updates automatically — no changes to `sidebars.ts` are needed

**Example: adding a new integration guide**

```bash
# Create the file
touch docs/docs/user-guides/admin/integrations/shopify-integration.md
```

```markdown
---
title: Shopify Integration
description: Connect your Shopify store to automatically create service records from orders.
sidebar_position: 2
---

# Shopify Integration

...
```

## Adding a New Top-Level Section

1. Create a new folder under `docs/docs/`
2. Add a `_category_.json` file in the folder:

```json
{
  "label": "Section Label",
  "position": 6,
  "link": {
    "type": "generated-index",
    "description": "One sentence describing what this section contains."
  }
}
```

3. Add `.md` files with the correct frontmatter

---

## Category Metadata (`_category_.json`)

Every folder must have a `_category_.json` to control its label and position in the sidebar.

| Field | Purpose |
|---|---|
| `label` | Displayed name in the sidebar |
| `position` | Integer order relative to sibling sections |
| `link.type` | Use `"generated-index"` to auto-generate a category landing page |
| `link.description` | Short description shown on the generated index page |

## Writing Style Guide

Follow these conventions for consistent, professional documentation:

### Voice and Tone
- Write in **second person** (you, your): "You can configure billing in Service Options."
- Use **present tense**: "PartnerFlow generates the invoice automatically."
- Use **active voice**: "Click Save" not "Save should be clicked."
- Be direct and concise — avoid padding phrases like "Please note that..." or "It is important to..."

### Headings
- Use `##` for major sections within a page
- Use `###` for subsections
- Headings should be **action-oriented** or **noun phrases**: "Creating a Stage", "Transition Configuration", not "How to Create a Stage"

### Lists
- Use bullet lists for items without a strict order
- Use numbered lists for sequential steps
- Keep list items parallel in structure

### Code and UI References
- Wrap UI elements in bold: **Settings → Users**, **New Service**, **Save**
- Wrap code, file names, and field names in backticks: `openapi.yaml`, `sidebar_position`
- Use code blocks with language tags for multi-line code examples

### Tables
Use tables to compare options or present structured reference data:

```markdown
| Option | Behaviour |
|---|---|
| Never | No charges applied |
| Sometimes | Charges on specific paths |
| Always | Every record generates an invoice |
```

### Admonitions
Use Docusaurus admonitions sparingly for genuinely important callouts:

```markdown
:::note
This feature is for internal and service provider use only. Customers cannot access it.
:::

:::warning
Deleting a service does not delete existing service records.
:::
```

## API Documentation

The `docs/api/` folder is **entirely auto-generated** from `openapi.yaml`. Do not edit files in this folder manually — they will be overwritten the next time the CI runs.

To update API documentation:
1. Update the OpenAPI spec in the main application (`warranty-flow-portal`)
2. The `openapi-sdk.yml` GitHub Actions workflow will automatically:
   - Generate the new `openapi.yaml`
   - Copy it to this repo
   - Run `npm run gen-api-docs` to regenerate all MDX files
   - Commit and push the changes

The `docs/api/` folder is gitignored — it does not exist in the repository and is created fresh on every deploy. To generate it locally for development:

```bash
npm run gen-api-docs   # writes docs/api/ from openapi.yaml
npm start              # dev server will now include the API reference
```

## Instructions for AI Agents

When an AI agent is asked to add or update documentation:

### Adding Documentation for a New Feature

1. Identify where the feature fits in the section table above
2. Create a `.md` file in the correct folder with the required frontmatter
3. Follow the style guide — second person, present tense, action-oriented headings
4. Link to related pages using relative doc paths: `[Service Options](/docs/create-service/service-options)`
5. Set `sidebar_position` to place the page logically (check the positions of neighbouring files)

### Updating Existing Documentation

1. Read the existing file before making changes
2. Preserve the frontmatter structure
3. Keep the same heading hierarchy and writing style as the existing content
4. Update internal links if page paths change

### Updating Release Notes

Add a new section at the top of `docs/release-notes.md` following this format:

```markdown
## v[VERSION] — [Month Year]

### [Feature Area]
- [What was added/changed/fixed]
```

### What Not to Touch

- `docs/api/` — auto-generated, never edit manually
- `docs/docusaurus.config.ts` — only modify if adding a new plugin or changing global config
- `docs/sidebars.ts` — autogenerated configuration, do not add doc IDs manually
- `openapi.yaml` — auto-updated from the main app repo
